# 数据飞轮 (Data Flywheel) 发布规范

本文件定义了将 Obsidian 笔记或用户对话转化为 "数据飞轮" 交互页面的核心约束。

## ⚠️ 强限制 (Hard Constraints)
**严禁随意发挥。** 任何记录在 "数据飞轮" 页面中的条目必须满足以下 **双重锚定**：

## 📅 阶段性锁定：内容驱动 (Phase Lock: Content-Driven Only)
**前端开发已进入观察期/锁定状态。** 在完成 Obsidian 库的批量迁移和内容深度润色之前，不允许进行非必要的前端交互页面开发或 UI 指令执行。

1.  **来源锚定 (Source Anchor)**：必须提供明确的来源文件链接（如 `obsidian/灵感/xxx.md`）或对话历史锚点。
2.  **原话锚定 (Literal Anchor)**：必须引用用户的原话片段。在进行结构化或哲学关联时，只能对原意进行升华，不得偏离或凭空捏造事实。

## 🛠 发布流程 (Publishing Workflow)

1.  **检索 (Retrieve)**：首先使用 `grep_search` 或 `read_file` 在工作区中定位原始素材。
2.  **提取 (Extract)**：摘录原话（Quote），记录原始路径（Path）。
3.  **构造 (Structure)**：
    *   `观察 (Observation)`: 直接引用的用户原话。
    *   `关联 (Philosophy/Logic)`: 基于原文的逻辑推演或哲学范式映射。
    *   `来源 (Context)`: 指向具体的本地文件路径。

## 🚫 负面约束 (Negative Constraints)
*   **不准** 在没有找到对应 `.md` 文件或具体对话记录的情况下，根据关键词（如"冷水澡"）自行联想细节。
*   **不准** 使用 AI 生成的虚构案例填充"数据飞轮"的交互卡片。
*   **不准** 忽略用户对"贴近原话"的要求。

## 🔄 修正案：冷水澡条目处理
如发现条目（如 `project-cold-shower`）缺乏明确的来源支撑，必须重新检索用户笔记库中的相关关键词。若库中无记录，则应将其标记为 `Draft/Unofficial` 或在用户提供明确原话后再行发布。

# 书籍碰撞流程 (Book Collision Protocol)

本文件定义了将书籍与个人哲学洞见库进行碰撞、并将有效碰撞升级为洞见卡的完整流程。

---

## 🎯 触发场景

当用户发送以下任意一种输入时，启动本流程：

- 书名 + 读后感受（哪怕只有一句话）
- 书名 + 摘录片段
- 书名 + "帮我碰撞"

**不要**在用户只是提到书名但未表达任何感受或想法时主动启动碰撞。
**不要**在用户明确说"只是记录读过"时生成碰撞分析。

---

## 📐 流程结构

### Step 1：生成书籍快照卡

读取用户输入，提炼以下三项，**不得超过这三项**：

- **核心主张**：这本书最想说的一件事（一句话）
- **预设读者**：这本书默认你是谁、有什么问题
- **与你的距离**：这本书的出发点和你的认知起点，是接近还是有张力

快照卡写入：
`~/Documents/codeStore/obsidian/阅读/{日期}-{书名}.md`

格式如下：

```
---
书名：{书名}
作者：{作者}
读完日期：{日期}
状态：已读 / 在读 / 放弃
---

## 核心主张
{一句话}

## 预设读者
{一句话}

## 与我的距离
{一句话，接近 / 互补 / 张力}

## 原始感受（用户原话）
{保留用户原话，不做修饰}
```

---

### Step 2：碰撞扫描

将书籍快照与用户现有哲学洞见库进行对比，输出三类碰撞：

#### A. 强共鸣点
你已有这个认知，书给了更清晰的语言或更坚实的论据。
→ 输出：已有哪张洞见卡 + 书的哪个表述可以加固它

#### B. 互补点
你和书各占一半，单独都不完整，合并后产生新的认知。
→ 输出：你的认知边界在哪 + 书补充了什么 + 合并后的新陈述是什么

#### C. 真正的张力点
你的认知与书存在实质分歧，不是表达不同，而是逻辑方向相反。
→ 输出：分歧的核心命题是什么 + 你的立场 + 书的立场 + 你是否愿意修正

**严格禁止**：
- 不准把表达风格差异误判为张力
- 不准为了输出"张力"而强行制造对立
- 不准在用户未提供洞见库内容时凭空捏造碰撞

---

### Step 3：人工裁决门禁（Human-in-the-loop）

碰撞扫描完成后，**停下来**，向用户呈现：

```
碰撞扫描完成。

【强共鸣】{N} 个
【互补点】{N} 个
【张力点】{N} 个

哪些你想升级为洞见卡？
哪些只是"读过，知道了"？
哪些你想反驳？
```

**不要**在用户确认前自动生成洞见卡。
**不要**把所有碰撞点都推荐升级，默认只有互补点和张力点值得升级。

---

### Step 4：升级为洞见卡（按需执行）

仅对用户确认的碰撞点执行。

读取 `INSIGHT-CARD-TEMPLATE.md` 获取卡片结构约束，生成洞见卡。

升级时的额外约束：

- **来源字段**必须同时标注书籍来源和你的原始洞见来源
- **冗余过滤器**中的噪音，优先使用书中被你过滤掉的表述（而非泛泛陈词）
- **对话还原**至少有一组 Q&A 体现"书的视角 vs 你的视角"的交锋
- **哲学笔记**模型名称格式：`{书名} × {你的洞见标题}`，体现碰撞来源

写入路径遵循 `INSIGHT-CARD-TEMPLATE.md` 中的落地规则。

---

### Step 5：精准推荐（按需执行）

触发条件：用户主动问"推荐我读什么" **或** 碰撞完成后用户说"还有什么适合我读"。

推荐逻辑：

1. 扫描现有洞见库，找出**认知空白维度**——有感受但缺语言、有直觉但缺框架的区域
2. 从已读书籍快照卡中找互补书（同一维度的不同角度）
3. 从未读书籍中推荐，必须说明：这本书填你哪个空白，而不是"你可能喜欢"

推荐格式：

```
你在"{洞见维度}"上有强烈的个人认知，但缺少{框架/语言/反例}。

推荐：《{书名}》
理由：它的核心主张是"{一句话}"，
      恰好从{角度}切入你已有的"{洞见卡标题}"。
```

**严格禁止**：
- 不准基于书籍类型或风格相似性推荐
- 不准一次推荐超过 3 本
- 不准在没有扫描洞见库的前提下给出推荐

---

## 📂 文件落地总览

| 内容 | 路径 |
|------|------|
| 书籍快照卡 | `~/Documents/codeStore/obsidian/阅读/{日期}-{书名}.md` |
| 碰撞原始记录 | 追加写入快照卡末尾的 `## 碰撞记录` 区块 |
| 升级后的洞见卡 | 遵循 `INSIGHT-CARD-TEMPLATE.md` 落地规则 |

---

## 🚫 全局禁止

1. 不准在用户未确认前自动执行 Step 4
2. 不准把书的观点直接写入洞见卡，必须经过碰撞、有你的原始认知作为锚点
3. 不准用书的语言覆盖你的原话，原话永远优先
4. 不准生成"这本书很好，值得一读"类型的书评式总结
5. 不准把"读过"等同于"碰撞完成"

# 洞见卡模板 (Insight Card Template)

本文件定义了哲学飞轮页面中每张洞见卡的完整 HTML 结构和内容约束。

---

# 复盘飞轮卡片约束 (Retrospective Flywheel Card)

本文件定义了将阶段性复盘材料压缩为"飞轮卡片"的核心约束。
飞轮卡片用于支持后续人机协作、人工判断和流程复用。

---

## 🎯 触发场景

当任务进入以下场景之一时，应生成飞轮卡片：

- 一个阶段性工作完成，用户提供"阶段复盘 / 事件级复盘 / 阶段思考"类 Markdown 文本
- 任务中明显体现 `PDCA`、`版本迭代`、`人工门禁`、`样本先行再放量` 等高杠杆思想
- 用户要求"提炼思维模型 / 模式 / 反模式 / 活人感 QA / 飞轮卡片"

**不要**把输出写成冗长的流程说明、知识库总览或零碎标签列表。
必须优先把材料压缩为一张飞轮卡片。

---

## 📐 飞轮卡片结构

当用户要求生成飞轮卡片时，按以下 5 个部分组织：

### 1. 这次实践为什么强烈体现某个思想标签
- 仅允许少量高密度思想标签，如：`PDCA`、`版本迭代`、`人工门禁`、`样本验证后放量`
- 说明这次事件为什么强烈体现该思想
- 指出它解决的核心问题是什么

### 2. 这个思想在这次事件里具体落在哪
- 落到具体的人工判断点，不是抽象描述
- 明确：人在哪一步介入，看什么、判断什么、依据什么放行或打回

### 3. 这类实践里最值得继续挖的人工判断是什么
- 明确指出思想落地过程中的"人工判断接口"
- 人必须在哪一步介入，判断的输入和输出分别是什么

### 4. AI 可以怎么通过提问辅助这段人工行为
- AI 不替代人工审批，只能辅助提问、整理和记录
- 优先输出三类问题：
  - **审批前**：暴露未收敛点的问题
  - **审批时**：帮助形成放行标准的问题
  - **审批后**：记录判断依据和风险的问题

### 5. 这张卡片想留给未来的启发
- 不是复述流程发生了什么
- 而是指出：以后在什么信号出现时，应再次调用这张卡片的思想

---

## 🔗 与 Phase 0 门禁的关系

若当前任务处于批量创建、公共步骤抽取、状态矩阵分析等高风险阶段，
飞轮卡片是对 Human-in-the-loop 的增强补充：

- Phase 0 负责阻断和等待人工审批
- 飞轮卡片负责把"人工为什么要介入、介入时该问什么"结构化出来

**飞轮卡片不是替代门禁，而是把门禁变成更顺畅、更可测的人机协作接口。**

---

## 🚫 严格禁止

1. 不要输出大量细碎标签
2. 不要把飞轮卡片写成 SOP 大全
3. 不要把飞轮卡片写成纯抽象感悟，必须落到具体人工判断点
4. 不要把 AI 写成自动裁决者，AI 只能辅助提问、整理和记录

---

## 📂 落地与同步规则

当用户发送复盘文档或洞见文本时，执行以下自动流程：

### Step 1: 生成飞轮卡片
- 读取 `.rules/RETROSPECTIVE-FLYWHEEL-CARD.md`（本文件）获取约束
- 读取 `.rules/INSIGHT-CARD-TEMPLATE.md` 获取哲学洞见卡模板（如同时产生洞见）
- 基于用户原文生成结构化卡片

### Step 2: 落地到 Obsidian 仓库
- 将原始复盘内容写入 `~/Documents/codeStore/obsidian/工作/{日期}-{标题}.md`
- 保留用户原话，不做 AI 风格修饰
- 自动 commit + push

### Step 3: 落地到交互仓库
- 在 `regeorge.github.io/` 中创建对应的交互页面（如尚未存在）

## 📅 阶段性锁定：内容驱动 (Phase Lock: Content-Driven Only)
**前端开发已进入观察期/锁定状态。** 在完成 Obsidian 库的批量迁移和内容深度润色之前，不允许进行非必要的前端交互页面开发或 UI 指令执行。

1.  **来源锚定 (Source Anchor)**：必须提供明确的来源文件链接（如 `obsidian/灵感/xxx.md`）或对话历史锚点。
2.  **原话锚定 (Literal Anchor)**：必须引用用户的原话片段。在进行结构化或哲学关联时，只能对原意进行升华，不得偏离或凭空捏造事实。

## 🛠 发布流程 (Publishing Workflow)

1.  **检索 (Retrieve)**：首先使用 `grep_search` 或 `read_file` 在工作区中定位原始素材。
2.  **提取 (Extract)**：摘录原话（Quote），记录原始路径（Path）。
3.  **构造 (Structure)**：
    *   `观察 (Observation)`: 直接引用的用户原话。
    *   `关联 (Philosophy/Logic)`: 基于原文的逻辑推演或哲学范式映射。

# SKILL: Obsidian 笔记转作品集工作流 (Obsidian Note-to-Portfolio Workflow)

**适用场景**：将 Obsidian 仓库中的笔记（带图片）转换为 `regeorge.github.io` 仓库中的可视化交互作品。
**核心价值**：自动化处理过程中的图片压缩、路径转换与导航同步。

---

## 🛠️ 核心操作规范 (SOP)

### 1. 内容迁移与解析 (Source Processing)
-   **提取逻辑**：从 Obsidian 目录提取 `[Title].md`，识别其中的 Markdown 图片链接 `![alt](path/to/img)` 或 Obsidian 语法 `[[img.jpg]]`。
-   **关联性**：确保迁移后的图片能够与文章建立逻辑关联。

### 2. 自动化资产压缩 (Auto-Compression)
-   **原则**：原始照片通常在数 MB 以上，严禁直接上传至 GitHub Pages。
-   **处理目标**：
    -   **格式转换**：优先转换为 `.webp` 或高压 `.jpg`。
    -   **尺寸限制**：宽度最大宽度设为 `1920px`。
    -   **体积控制**：单张图片控制在 `200KB` 以内。
-   **存放位置**：统一维护在 `regeorge.github.io/assets/img/[project-name]/`。

### 3. 作品转换模版 (HTML Generation)
-   **结构转换**：将 Markdown 转换为带 Claude/Obsidian 风格的 HTML 入口。
-   **路径动态化**：将 HTML 中的 `img` 标签自动指向 `assets/img/` 下的压缩版。

### 4. 自动化脚本调用 (Command Runner)
-   建议封装为 `publish-note.py` 脚本，执行以下逻辑：
    1.  读取 `obsidian/` 路径下的指定 MD。
    2.  利用 Python 库 (如 `Pillow`) 批量处理图片并输出至 `assets/`。
    3.  生成 HTML 文件夹并更新 `regeorge.github.io/index.html` 的导航项。

---

## 🎨 视觉语言强制要求

-   **Alt 文本**：所有压缩后的图片必须保留 Markdown 中的 Alt 描述以增强 SEO。
-   **占位符**：对于大图，优先生成一个小尺寸的 Base64 模糊占位符。
-   **响应式**：图片容器必须支持 `max-width: 100%; height: auto;`。

---

## 📂 目录结构约束

```text
regeorge.github.io/
├── assets/
│   └── img/
│       └── [project-name]/  <-- 所有该文章的压缩图存放在此
├── [project-name]/
│   └── index.html           <-- 作品展示页
```

# Obsidian 笔记记录规范 (AI Writing Protocol)

本规范定义了 AI 在响应用户"帮忙记录"指令，并向 Obsidian 笔记库执行写入操作时的核心约束。

## ⚠️ 强约束：原话优先 (Verbatim-First Constraint)
**严防语义漂移。** 笔记内容必须深度锚定用户的原始输入。

## 📍 当前阶段：内容迁移与深度对话 (Current Phase: Content Migration & Deep Dialogue)
**当前任务重心不在前端，而在 Obsidian 库的构建。**
1. **全面迁移**：优先协助用户将分散的笔记碎片迁移至 Obsidian 对应的分类目录中。
2. **深度润色**：基于迁移后的海量数据，与用户进行深度的逻辑对谈。在深聊过程中，通过 AI 的结构化能力，对笔记内容进行无损且有深度的润色。
3. **前端冻结**：在笔记内容未达到"深度润色完成"状态前，严禁大规模更新前端交互页面（如数据飞轮 Index）。

## 🛠 写入工作流 (Writing Workflow)

1.  **捕获**：识别用户指令中的"原子化观察"。
2.  **镜像**：在草稿中重现用户原话，确保核心逻辑无损。
3.  **塑形**：应用以下结构：
    *   `观察 (Observation)`: 直接引用的用户原话。
    *   `元数据 (Metadata)`: 自动添加标签（Tags）和相关链接（Links）。
    *   `思考 (Reflection)`: 若用户要求，可基于原意进行哲学关联，但需与原话明确区分。

*   **不准** 在用户未提供具体案例时，自行生成"例如"、"比如"。

*   **不准** 忽略用户的特定表达风格。
